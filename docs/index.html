01 · Overview

Built for the spaces the cloud can't reach.

Organizations operating in secure environments face a common challenge: knowledge is scattered across formats, teams, and tools — and existing AI solutions demand exactly the cloud connectivity those environments forbid. Vāgmi was originally designed for deployment within air-gapped facilities such as DRDO sites, where privacy, local processing, and user isolation are non-negotiable.

Scattered Documents

Files spread across PDFs, DOCX, and TXT — no unified search or grounding layer for AI.

Siloed Knowledge

Critical context trapped in individual teams, with no shared intelligence substrate.

Cloud-Dependent AI

Existing assistants route prompts through external APIs — incompatible with restricted networks.

Connectivity Lock-In

Collaboration tools assume always-on internet; air-gapped teams are left without options.

Manual Diagramming

Generating flowcharts and structured content is repetitive, time-consuming, and disconnected from sources.

Privacy Constraints

Every byte of data must remain on local infrastructure — no exfiltration, no exceptions.

Vāgmi addresses all of these through a centralized offline intelligence platform that runs entirely on local infrastructure. Upload documents, chat with a multi-agent AI grounded in your own files, generate Mermaid diagrams, and collaborate with teammates through rooms and direct messages — all packaged as a native desktop app with zero external network dependency.
02 · High-Level Architecture

Three layers. One localhost. Zero internet.

A PyWebView desktop client talks to a local FastAPI backend over 127.0.0.1, backed by a service layer of singletons covering the hybrid RAG pipeline, multi-agent system, and LLM service. Storage is split across SQLite, ChromaDB, per-user filesystem directories, and locally-stored models. Nothing in the loop touches the internet.

04 · Authentication & Access Control

Owner bootstrap. Admin gate. JWT everywhere.

The first user to register on a deployment is automatically made the workspace owner — approved immediately, workspace provisioned on the spot. Every subsequent registration is created with is_approved=false and must be approved by an admin before they can log in.

05 · Retrieval Stack

Hybrid retrieval. Vector + BM25. Weighted fusion.

The retrieval layer runs automatically whenever a document is uploaded. Extraction → chunking → embeddings → dual indexing (ChromaDB + BM25). At query time, both retrievers fire, scores are normalized, fused with weights 0.4 × BM25 + 0.6 × Vector, and the deduplicated top-k chunks are returned.

STAGE 01

Document Upload

POST /documents/upload — accepts .pdf, .docx, .txt up to 1 GB. Validated, then saved to storage/users/user_id/documents/uuid.ext with a Document row marked status = 'processing'.

STAGE 02

Text Extraction

DocumentProcessor.extract_text — PyMuPDF (fitz) for PDF pages, python-docx for DOCX paragraphs, UTF-8 read for TXT.

STAGE 03

Chunking

ChunkingService.chunk_text — sliding-window algorithm, chunk_size=500 chars, overlap=100 chars.

STAGE 04

Embeddings

EmbeddingService.embed_chunks — all-MiniLM-L6-v2, 384-dim, normalized. Runs with local_files_only=True for fully offline inference.

STAGE 05

Dual Indexing

ChromaDB: persistent document_chunks collection.
BM25: BM25Plus index persisted to per-user bm25_index.json.

STAGE 06

Query & Fusion

ChromaDB returns top k × 4; BM25 returns top k. Normalized, fused (0.4 BM25 + 0.6 Vector), deduplicated by chunk_text, sorted by final score.

06 · Multi-Agent AI Layer

Five specialist agents. One orchestrator.

The intelligence module runs a fully offline, session-based multi-agent system on top of the retrieval stack. Each agent is defined as an AgentSpec with its own name, max tokens, temperature, artifact type, and purpose. Routing supports manual selection or auto-detection via a keyword bank.

Master Agent
Asks a clarifying question when intent is ambiguous
max_tokens: 192 temp: 0.2 artifact: None

The Master agent never produces a substantive answer. It is invoked when routing confidence falls below 0.55, or when no keyword matches are found. Instead, it returns a clarification reply with suggestion labels — letting the UI offer the user a one-tap choice between likely specialist agents.

Query Agent
Direct answers grounded in selected documents
max_tokens: 1024 temp: 0.25 artifact: None

The workhorse agent for everyday questions. Grounded in the session's selected documents and the last 10 messages of conversation context. Matched by keywords like what, who, how, explain. Confidence base for auto-routing: 0.84.

Summary Agent
Structured summaries with key ideas and takeaways
max_tokens: 1536 temp: 0.3 artifact: summary

Produces structured recaps — triggered by keywords like summary, recap, tl;dr. Higher token budget allows for multi-section output. Generates a persistent summary artifact saved to the user's workspace. Confidence base: 0.92.

Document Agent
Long-form, polished markdown drafts
max_tokens: 2048 temp: 0.35 artifact: document

The highest-budget agent — for reports, drafts, proposals. Matched on keywords like report, draft, proposal. Slightly higher temperature (0.35) gives it latitude for prose quality while staying grounded in retrieved chunks. Generates a document artifact. Confidence base: 0.88.

Diagram Agent
Mermaid flowcharts and sequence diagrams
max_tokens: 1024 temp: 0.2 artifact: mermaid

Outputs Mermaid-syntax diagrams rendered live in the desktop client's diagram studio. Low temperature (0.2) keeps syntax valid. Matched by keywords like diagram, mermaid, flowchart. Generates a mermaid artifact exportable to SVG/PNG. Confidence base: 0.90.

Orchestration Flow

AiOrchestrator.run_session_turn drives a full turn in six steps:

STEP 01

Route

Determine the agent per routing rules — manual selection or auto keyword-bank match. Confidence and needs_clarification are set here.

STEP 02

Build Context

ContextService loads the session's selected documents and the last 10 messages for conversational continuity.

STEP 03

Assemble Prompt

HybridRetrievalService pulls up to 12 grounding chunks (1200 chars each). PromptService assembles identity, metadata, guidance, contract, and passages.

STEP 04

Clarification Check

If needs_clarification is true, a clarification reply with suggestion labels is returned immediately — skipping generation entirely.

STEP 05

Generate Reply

LLMService ensures the worker is running, sends the prompt, waits up to 600s. On failure, falls back to a grounded non-LLM reply built from retrieved chunks.

STEP 06

Persist Turn

User message, assistant reply, and any generated artifact are saved. Session last_prompt and status updated before the full payload returns to the UI.

Request Proxying

Custom VagmiRequestHandler inspects every HTTP method. /api/* requests are forwarded to FastAPI with hop-by-hop headers stripped and 64KB chunked streaming.

SPA Architecture

splash.html performs the token check, then loads index.html — a sidebar + iframe SPA routing between home, intelligence, chat, admin, settings, and diagram pages.

Mini-Dock Mode

A separate 320×460 frameless, always-on-top window. enter_compact_mode hides the main window; closing the mini-dock restores it with page state preserved.

Native Bridges

DesktopBridge exposes save-as dialogs, desktop notifications (notify-py), and voice recording control to the frontend via window.pywebview.api.





10 · Technology Stack

Batteries included. Cloud excluded.

Every dependency runs locally. No telemetry, no CDN, no external model APIs.

Backend
  • FastAPI + Uvicorn — async API server
  • SQLAlchemy + SQLite (aiosqlite) — 16 tables in vagmi.db
  • Pydantic / pydantic-settings — schemas & config
  • python-jose + passlib/bcrypt — JWT auth
  • python-multipart — file uploads
Retrieval & AI
  • ChromaDB — persistent vector store
  • sentence-transformers — all-MiniLM-L6-v2 (384-dim, offline)
  • rank-bm25 — keyword retrieval (BM25Plus)
  • PyMuPDF (fitz) — PDF extraction
  • python-docx — DOCX extraction
  • llama-cpp-python — local GGUF inference (Qwen2.5 quantized)
  • torch, transformers — embedding backend support
Desktop Client
  • PyWebView — native window + local HTTP server bridge
  • Vanilla JS, HTML, CSS — no frontend framework
  • Mermaid.js — in-app diagram rendering
  • notify-py — native desktop notifications
  • sounddevice / soundfile — voice message recording

12 · Getting Started

Up and running in two commands.

The server creates its SQLite tables automatically on startup. The first user to register becomes the workspace owner. Visit http://localhost:8000/docs for the Swagger UI.

1. Run the backend

cd backend

# install the full, pinned dependency set
pip install -r requirements-lock.txt

# place a compatible GGUF model in the configured
# local models directory, or point VAGMI_AI_MODEL_PATH at it

# run the server
uvicorn app.main:app --reload

2. Run the desktop client

cd desktop

pip install -r requirements-lock.txt

python main.py

# Register your first user — they become the
# workspace owner. Subsequent users need approval.
That's it. The backend auto-creates SQLite tables on startup. No external services, no API keys, no cloud credentials. The whole stack runs on a single machine.
